# Zuhui PPT QA

## Goal

Stop unverified slide decks from being delivered. A deck is not complete until PPTX, HTML, and PDF pass structural and content checks, and visual limitations are documented.

## Project Parameters

All paths use the project parameters defined in `zuhui-ppt-master`:

- `PROJECT_SLUG` — paper short name for file naming
- `VERSION` — current version number
- `OUTPUT_DIR` — `ppt_v{VERSION}/`
- `VISUAL_REF` — path to prior approved deck version, or `none`

## Required Checks

### Package Checks

- PPTX exists: `{OUTPUT_DIR}/{PROJECT_SLUG}_v{VERSION}.pptx`
- HTML exists: `{OUTPUT_DIR}/{PROJECT_SLUG}_v{VERSION}.html`
- PDF exists: `{OUTPUT_DIR}/{PROJECT_SLUG}_v{VERSION}.pdf`
- Speaker notes exist when planned.
- QA report exists.
- Assets folder exists with all referenced images.

### Count Checks

- PPTX slide count equals expected slide specs.
- HTML real slide count equals expected count.
- PDF page count equals expected count.
- PPTX slide XML count matches PPTX slide count.
- PPTX media count is nonzero when figure slides exist.

### Swiss International Checks

- Every real HTML slide has `data-layout`.
- Registered body layouts use S01-S22; cover/closing may use allowed `SWISS-*` extensions.
- Deck rhythm includes dark statement/chapter pages and light evidence pages.
- More than 25 slides should use enough layout variety, typically at least 8 layout IDs.
- All sans-serif fonts (no serif on Swiss decks).
- Single accent color per deck (no mixing IKB + yellow + orange, etc.).
- No gradients, shadows, or border-radius on any element.
- Typography weight ladder respected: larger text uses lighter weight.

### Content Density Checks

- No non-transition slide should be visually or semantically empty.
- Dense paper figures/tables should not be tiny.
- Figure-heavy pages should have a dominant visual and a narrow interpretation layer.
- Tables and appendices should appear when the paper is content-rich.
- Tables should be native/rebuilt when values are legible, not screenshots.
- Screenshots must not include long body paragraphs, figure legends, table captions, or surrounding PDF text.
- If `VISUAL_REF` points to a prior approved version, rendered QA must compare against that visual gold source rather than only checking structural validity.

### Banned Phrase Checks

Search slide text and HTML for:

- standalone English labels such as `WHY NOW`, `Q&A`, `controlled evaluation`, `autonomous research harness`, `components`, `claim`, or `workflow` unless paired as `中文（English）`
- glossary-only pages
- meta-description pages about deck production
- "why this paper is worth presenting" or "how to read this deck"

Add any user-specific banned phrase from feedback.

### Evidence Checks

- No unsupported invented metrics.
- Observational evidence is labeled as observational.
- AI reviewer scores are not presented as human evaluation.
- Limitations are included.
- Figures/tables have source labels.

## Rendered QA

**Standard visual review pipeline** (default):

1. **HTML → PDF**: Use headless browser (Chrome/Edge headless) to print the HTML to PDF (`@page { size: 13.333in 7.5in; margin: 0; }`)
2. **PDF → per-page images**: Use PyMuPDF to split the PDF into individual PNG images, one per slide (1920x1080 resolution)
3. **Per-page visual review**: Review each page image individually for:
   - image size, text overflow, missing images, rhythm, and page-2/title overlap,
   - whether figure crops are clean image-only crops,
   - whether visible English appears without immediate Chinese translation,
   - repair weak slides.

If a reliable renderer is available:
- Follow the standard pipeline above.
- Create a contact sheet from the per-page images.

If no renderer is available:
- do package-level validation,
- inspect extracted assets/contact sheets,
- explicitly report that pixel-level rendered QA was not performed.

## QA Report Contents

Write a concise report in `{OUTPUT_DIR}/qa_report_v{VERSION}.md` with:

- output paths,
- source files,
- slide/page counts,
- media count,
- layout compliance,
- banned phrase result,
- figure asset result,
- rendered QA status,
- PPTX editability status, especially if visual-fidelity screenshots were used,
- known limitations,
- comparison against `VISUAL_REF` (if applicable).

## Delivery Rule

Never say "done" unless the QA report exists and the command output or script result confirms the checks.

After QA passes, run `zuhui-ppt-visual-score` to get the visual quality signal. Both QA and visual score must be documented before the deck can be marked as delivered.
