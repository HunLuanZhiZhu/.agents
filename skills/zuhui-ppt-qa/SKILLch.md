# Zuhui PPT QA（中文说明）

## 目标

防止未验证的 PPT 交付。PPTX、HTML、PDF 都通过结构和内容检查后，才算完成。

## 必查项目

### 产物检查

- PPTX 存在；
- HTML 存在；
- PDF 存在；
- 讲稿存在；
- QA 报告存在；
- assets 文件夹存在且图片齐全。

### 数量检查

- PPTX 页数等于预期；
- HTML 真实 slide 数等于预期；
- PDF 页数等于预期；
- PPTX slide XML 数正确；
- 有图页时 PPTX media 数不能为 0。

### Guizang Swiss 检查

- 每个真实 HTML slide 有 `data-layout`；
- 正文页使用 `S01–S22`，封面/尾页可用允许的 `SWISS-*`；
- 有暗色 statement/章节页和白底证据页；
- 25 页以上通常至少 8 种 layout ID。

### 内容密度检查

- 非过渡页不能空；
- 论文图表不能太小；
- 图表页应让图占主导，解释栏较窄；
- 内容丰富论文要包含表格、附录和实现细节。
- 表格只要数值可读，就应原生重建，不能截图糊弄；
- 截图不能包含大段正文、图注、表注或 PDF 周围文字；
- 如果用户更喜欢某个 v1 版本，渲染 QA 要和这个视觉金标准对照，而不只是看结构是否通过。

### 禁用词检查

搜索：

- `中文翻译（English term）`
- `图里全是英文`
- `这版怎么解决`
- `术语记忆`
- `为什么这篇文章值得`
- `Claim–Evidence`
- 裸露英文标签，例如 `WHY NOW`、`Q&A`、`controlled evaluation`、`autonomous research harness`、`components`、`claim`、`workflow`，除非写成 `中文（English）`

以及用户额外指出的禁用表达。

### 证据检查

- 不编造指标；
- 观察性证据要标注；
- AI 审稿分不能当人类评测；
- 必须包含局限；
- 图表要有来源。

## 渲染 QA

如果有渲染器：

- 渲染 PDF 或截图；
- 生成 contact sheet；
- 检查图片大小、文字溢出、缺图、节奏，以及第 2 页/标题是否重叠；
- 检查图片裁剪是否为干净纯图，不带论文正文和 caption；
- 检查 PPT 可见英文是否都有即时中文翻译；
- 修复弱页。

如果没有渲染器：

- 做包结构验证；
- 检查资产 contact sheet；
- 明确说明没有做像素级渲染 QA。

## QA 报告

报告应包含：

- 输出路径；
- 源文件；
- 页数；
- media 数；
- layout 合规；
- 禁用词；
- 图片资产；
- 渲染 QA 状态；
- PPTX 可编辑性状态，尤其是使用视觉保真截图型 PPTX 时；
- 已知限制。

## 交付规则

没有 QA 报告和验证输出，不要说“完成”。
